# Hotel Management System - API Documentation

> **Base URL:** `http://localhost:8000` (or your configured host)
>
> **Architecture:** Server-rendered PHP application. All routes return HTML pages or perform 302 redirects. There is no JSON API -- every interaction is via standard browser form submissions and page navigation.
>
> **CSRF Protection:** All POST routes require a `csrf_token` field that matches the token stored in `$_SESSION['csrf_token']`. CSRF enforcement can be disabled globally via the app's security configuration (vulnerable mode). The token is a 64-character hex string generated by `bin2hex(random_bytes(32))`.
>
> **Authentication:** Session-based. A logged-in user has `$_SESSION['user_id']`, `$_SESSION['username']`, and `$_SESSION['role']` set. Routes that require login redirect unauthenticated users to `/login?redirect=<current_uri>`. Admin routes additionally check `$_SESSION['role'] === 'admin'` and return 403 if the user is not an admin.
>
> **Security Modes:** The application has a "secure" mode and a "vulnerable" mode (for cybersecurity education). In secure mode, passwords are hashed with bcrypt (cost 12), sessions are regenerated on login, and accounts lock after 5 failed attempts. In vulnerable mode, passwords may be stored/compared in plaintext and CSRF may be disabled.

### POST /contact

**Auth Required:** No (but if logged in, user ID is recorded in the activity log)
**CSRF Required:** Yes
**Description:** Processes a contact form submission. Validates inputs, sanitizes with `strip_tags()`, and logs the inquiry to the `activity_logs` table. Does not send an email -- records the inquiry for admin review.

**POST Fields:**

| Field      | Type   | Required | Validation                                               |
|------------|--------|----------|----------------------------------------------------------|
| name       | string | Yes      | Must not be empty. Sanitized with `strip_tags()`.        |
| email      | string | Yes      | Must be a valid email (`FILTER_VALIDATE_EMAIL`). Sanitized with `strip_tags()`. |
| phone      | string | No       | No format validation. Sanitized with `strip_tags()`.     |
| subject    | string | Yes      | Must not be empty. Sanitized with `strip_tags()`.        |
| message    | string | Yes      | Must be at least 10 characters (`mb_strlen`). Sanitized with `strip_tags()`. |
| csrf_token | string | Yes      | Must match session token.                                |

**Log Entry:** The full inquiry (name, email, phone, subject, message) is concatenated into a description string (max 1000 characters, truncated with `...`) and inserted into `activity_logs` with action `contact_inquiry`, entity_type `contact`, and severity `info`.

**Success Response:** 302 redirect to `/contact` with flash message: "Thank you for your message! We'll get back to you within 24 hours."

**Error Responses:**
- Invalid CSRF: 302 redirect to `/contact` with flash error "Invalid security token. Please try again."
- Validation failure: 302 redirect to `/contact` with flash error listing all validation errors (space-separated).

**Example:**
```bash
curl -X POST http://localhost:8000/contact \
  -d "name=John+Doe" \
  -d "email=john@example.com" \
  -d "phone=+91-9876543210" \
  -d "subject=Room+Inquiry" \
  -d "message=I+would+like+to+know+about+your+suite+rooms+availability+in+December." \
  -d "csrf_token=YOUR_CSRF_TOKEN" \
  -b "PHPSESSID=your_session_id"
```

Handles user login, registration, logout, and password reset. Sessions are PHP native sessions. In secure mode, session IDs are regenerated on login and accounts lock after repeated failures.

**Parameters:** None

**Already Logged In:** 302 redirect to `/`.

**Example:**
```bash

```

---

### POST /login

**Auth Required:** No
**CSRF Required:** Yes
**Description:** Authenticates a user. On success, sets session variables (`user_id`, `username`, `role`, `LAST_ACTIVITY`) and redirects to the requested page or homepage. On failure, flashes an error and redirects back to login. In secure mode, uses `password_verify()` against bcrypt hash; in vulnerable mode, does a direct string comparison. Accounts lock for 30 minutes after 5 failed attempts (secure mode only).

**POST Fields:**

| Field      | Type   | Required | Validation                                            |
|------------|--------|----------|-------------------------------------------------------|
| username   | string | Yes      | Must not be empty. Looked up via `UserRepository::findByUsername()`. |
| password   | string | Yes      | Must not be empty. Verified against stored hash/password. |
| csrf_token | string | Yes      | Must match session token.                             |

**Query Parameters (on the login page URL):**

| Parameter | Type   | Required | Description                                           |
|-----------|--------|----------|-------------------------------------------------------|
| redirect  | string | No       | URL to redirect to after successful login. Defaults to `/`. |

**Session Variables Set on Success:**
- `$_SESSION['user_id']` -- integer user ID
- `$_SESSION['username']` -- string username
- `$_SESSION['role']` -- string role (`user` or `admin`)
- `$_SESSION['LAST_ACTIVITY']` -- Unix timestamp

**Account Lockout (Secure Mode):**
- Failed login attempts are tracked via `failed_login_attempts` column.
- After the 5th consecutive failure, the account is locked for 30 minutes (`locked_until` column).
- Successful login resets the counter and updates `last_login` and `last_login_ip`.

**Success Response:** 302 redirect to the `redirect` query param value, or `/` if not provided.

**Error Responses:**
- Invalid CSRF: 302 redirect to `/login` with flash error "Invalid CSRF token".
- Missing fields: 302 redirect to `/login` with flash error "Username and password are required".
- User not found: 302 redirect to `/login` with flash error "Invalid credentials".
- Account locked: 302 redirect to `/login` with flash error "Account is locked. Please try again later."
- Wrong password: 302 redirect to `/login` with flash error "Invalid credentials" (or lockout message if threshold reached).

**Example:**
```bash
curl -X POST http://localhost:8000/login \
  -d "username=johndoe" \
  -d "password=MyPassword123" \
  -d "csrf_token=YOUR_CSRF_TOKEN" \
  -b "PHPSESSID=your_session_id" \
  -L
```

### POST /register

**Auth Required:** No
**CSRF Required:** Yes
**Description:** Creates a new user account. On success, the user is automatically logged in (session variables set) and redirected to the homepage. New users are always assigned the `user` role.

**POST Fields:**

| Field      | Type   | Required | Validation                                                    |
|------------|--------|----------|---------------------------------------------------------------|
| username   | string | Yes      | Minimum 3 characters (`strlen`). Must be unique (checked via `UserRepository::usernameExists()`). |
| email      | string | Yes      | Must be a valid email (`FILTER_VALIDATE_EMAIL`). Must be unique (checked via `UserRepository::emailExists()`). |
| password   | string | Yes      | Minimum 6 characters (`strlen`). Hashed with bcrypt cost 12 in secure mode; stored plaintext in vulnerable mode. |
| full_name  | string | Yes      | Must not be empty.                                            |
| phone      | string | No       | No validation rules. Stored as-is.                            |
| csrf_token | string | Yes      | Must match session token.                                     |

**User Record Created:**
- `role` = `user`
- `is_active` = `1`
- `email_verified_at` = current timestamp (auto-verified)
- `created_at` = current timestamp

**Success Response:** 302 redirect to `/` with flash message "Registration successful! Welcome!" and auto-login (session set).

**Error Responses:**
- Invalid CSRF: 302 redirect to `/register` with flash error "Invalid CSRF token".
- Validation errors: 302 redirect to `/register` with flash error containing all errors joined by `<br>`.
- Database failure: 302 redirect to `/register` with flash error "Registration failed. Please try again."

**Example:**
```bash
curl -X POST http://localhost:8000/register \
  -d "username=janedoe" \
  -d "email=jane@example.com" \
  -d "password=Secret123" \
  -d "full_name=Jane+Doe" \
  -d "phone=+91-9876543210" \
  -d "csrf_token=YOUR_CSRF_TOKEN" \
  -b "PHPSESSID=your_session_id" \
  -L
```

### POST /forgot-password

**Auth Required:** No
**CSRF Required:** Yes
**Description:** Processes a password reset request. If the email exists, generates a cryptographically secure 64-character hex token (valid for 1 hour) and stores it in the user record. For demo purposes, the reset link is flashed to the page instead of being emailed. The response is always the same regardless of whether the email exists (prevents email enumeration).

**POST Fields:**

| Field      | Type   | Required | Validation                                          |
|------------|--------|----------|-----------------------------------------------------|
| email      | string | Yes      | Must be a valid email (`FILTER_VALIDATE_EMAIL`).    |
| csrf_token | string | Yes      | Must match session token.                           |

**Token Details:**
- Generated by `bin2hex(random_bytes(32))` -- 64-character hex string.
- Stored in `password_reset_token` column.
- Expiry set to current time + 1 hour in `password_reset_expires` column.
- Action is logged to `activity_logs` with action `password_reset_requested`.

**Success Response:** 302 redirect to `/forgot-password` with flash message "If an account exists with that email, reset instructions have been sent." (always shown). If a matching user exists, also flashes `reset_link` containing `/reset-password?token=<token>`.

**Error Responses:**
- Invalid CSRF: 302 redirect to `/forgot-password` with flash error "Invalid CSRF token".
- Invalid email format: 302 redirect to `/forgot-password` with flash error "Please enter a valid email address."

**Example:**
```bash
curl -X POST http://localhost:8000/forgot-password \
  -d "email=john@example.com" \
  -d "csrf_token=YOUR_CSRF_TOKEN" \
  -b "PHPSESSID=your_session_id" \
  -L
```

### POST /reset-password

**Auth Required:** No
**CSRF Required:** Yes
**Description:** Processes the password reset. Validates the token (must exist, not expired), enforces a strict password policy, and updates the user's password (bcrypt cost 12). On success, clears the reset token/expiry, resets failed login attempts and lockout, and logs the action.

**POST Fields:**

| Field            | Type   | Required | Validation                                                    |
|------------------|--------|----------|---------------------------------------------------------------|
| token            | string | Yes      | Must match a valid, non-expired `password_reset_token` for a non-deleted user. |
| new_password     | string | Yes      | Min 8 characters. Must contain at least one uppercase letter, one lowercase letter, and one number. |
| confirm_password | string | Yes      | Must exactly match `new_password`.                            |
| csrf_token       | string | Yes      | Must match session token.                                     |

**Password Policy:**
- Minimum 8 characters (`strlen`)
- At least one uppercase letter (`/[A-Z]/`)
- At least one lowercase letter (`/[a-z]/`)
- At least one digit (`/[0-9]/`)

**On Success, User Record Updated:**
- `password` = bcrypt hash (cost 12)
- `password_reset_token` = NULL
- `password_reset_expires` = NULL
- `failed_login_attempts` = 0
- `locked_until` = NULL

**Success Response:** 302 redirect to `/login` with flash message "Password reset successful! You can now login."

**Error Responses:**
- Invalid CSRF: 302 redirect to `/login` with flash error "Invalid CSRF token".
- Missing token: 302 redirect to `/login` with flash error "Invalid reset token."
- Invalid/expired token: 302 redirect to `/forgot-password` with flash error "Invalid or expired reset link."
- Password policy failure: 302 redirect to `/reset-password?token=<token>` with flash error listing all policy violations joined by `<br>`.

**Example:**
```bash
curl -X POST http://localhost:8000/reset-password \
  -d "token=abc123def456..." \
  -d "new_password=NewSecure1" \
  -d "confirm_password=NewSecure1" \
  -d "csrf_token=YOUR_CSRF_TOKEN" \
  -b "PHPSESSID=your_session_id" \
  -L
```

## 3. Room Browsing

Public routes for browsing and viewing room details. No authentication required. Rooms are filtered to show only non-deleted, available, and operationally functional rooms (for the public listing).

---

### GET /rooms

**Auth Required:** No
**CSRF Required:** No
**Description:** Lists available rooms with optional filtering by type, date range, guest count, price range, and sorting. All filters are validated and sanitized -- invalid values are silently discarded (treated as empty). Results limited to 50 rooms. Only rooms that are non-deleted, available (`is_available = 1`), and operational (`maintenance_status = 'operational'`) are shown.

**Query Parameters:**

| Parameter  | Type    | Required | Validation                                                                       |
|------------|---------|----------|----------------------------------------------------------------------------------|
| type       | string  | No       | Must be one of: `single`, `double`, `suite`, `deluxe`, `presidential`. Invalid values ignored. |
| check_in   | date    | No       | Must be valid `Y-m-d` format (verified via `DateTime::createFromFormat`). Invalid values ignored. |
| check_out  | date    | No       | Must be valid `Y-m-d` format. Must be after `check_in`. Invalid values ignored.  |
| guests     | integer | No       | Must be a positive integer (`ctype_digit`, >= 1). Filters rooms with `max_occupancy >= guests`. Invalid values ignored. |
| min_price  | numeric | No       | Must be a non-negative number. Filters rooms with `base_price >= min_price`. Invalid values ignored. |
| max_price  | numeric | No       | Must be a non-negative number. Filters rooms with `base_price <= max_price`. Invalid values ignored. |
| sort       | string  | No       | Must be one of: `price_asc`, `price_desc`, `type`. Invalid values ignored. Default sort: `room_type ASC, base_price ASC`. |

**Date Availability Logic:** When both `check_in` and `check_out` are valid, rooms with existing non-cancelled, non-no-show bookings that overlap the requested date range are excluded. The overlap check uses: `booking.check_in < requested.check_out AND booking.check_out > requested.check_in`.

**Success Response:** 200 OK -- HTML page listing filtered rooms with filter controls.

**Example:**
```bash
curl "http://localhost:8000/rooms?type=deluxe&check_in=2026-05-01&check_out=2026-05-05&guests=2&sort=price_asc"
```

### POST /book

**Auth Required:** Yes (any logged-in user)
**CSRF Required:** Yes
**Description:** Creates a new room booking. Performs comprehensive server-side validation, calculates pricing server-side (never trusts client), checks for date conflicts with existing bookings, generates a unique booking reference, and logs the action.

**POST Fields:**

| Field            | Type    | Required | Validation                                                                  |
|------------------|---------|----------|-----------------------------------------------------------------------------|
| room_id          | integer | Yes      | Must be strictly numeric (`ctype_digit`). Room must exist and be available. |
| check_in         | date    | Yes      | Must be valid `Y-m-d` format. Must be today or a future date.              |
| check_out        | date    | Yes      | Must be valid `Y-m-d` format. Must be strictly after `check_in`. Stay must be <= 30 nights. |
| num_guests       | integer | Yes      | Must be strictly numeric (`ctype_digit`), >= 1, and <= room's `max_occupancy`. |
| special_requests | text    | No       | Max 500 characters (`mb_strlen`). Sanitized with `strip_tags()`. NULL if empty. |
| csrf_token       | string  | Yes      | Must match session token.                                                   |

**Price Calculation (Server-Side Only):**
```
nights        = check_out_date - check_in_date (in days)
base_total    = room.base_price * nights (rounded to 2 decimals)
gst_rate      = 18% if room.base_price >= 7500, else 12%
tax_amount    = base_total * gst_rate (rounded to 2 decimals)
total_price   = base_total + tax_amount (rounded to 2 decimals)
```

**Date Conflict Check:** The system queries `BookingRepository::findByRoomAndDateRange()` for existing bookings (excluding cancelled and no-show) that overlap the requested date range. If any conflicts exist, the booking is rejected.

**Booking Record Created:**
- `booking_reference` = auto-generated via `Booking::generateReference()`
- `status` = `confirmed`
- `payment_status` = `paid`
- `discount_amount` = `0`
- `created_by` = current user ID

**Success Response:** 302 redirect to `/booking/{id}/confirmation` with flash message "Booking confirmed! Your reference number is **{reference}**."

**Error Responses:**
- Not logged in: 302 redirect to `/login?redirect=<current_uri>`.
- Invalid CSRF: 302 redirect back with flash error "Invalid security token. Please try again."
- Invalid room_id: 302 redirect back with flash error "Invalid room selection."
- Invalid date format: 302 redirect back with flash error "Invalid date format. Please use YYYY-MM-DD."
- Past check-in: 302 redirect back with flash error "Check-in date cannot be in the past."
- check_out not after check_in: 302 redirect back with flash error "Check-out date must be after check-in date."
- Stay > 30 nights: 302 redirect back with flash error "Maximum stay is 30 nights. Please shorten your reservation."
- Invalid guest count: 302 redirect back with flash error "Number of guests must be at least 1."
- special_requests > 500 chars: 302 redirect back with flash error "Special requests must be 500 characters or fewer."
- Room not found/unavailable: 302 redirect to `/rooms` with flash error "Room not found or no longer available."
- Guests exceed capacity: 302 redirect back with flash error "Number of guests exceeds room capacity of {max}."
- Date conflict: 302 redirect back with flash error "Room not available for selected dates. Please choose different dates."

**Example:**
```bash
curl -X POST http://localhost:8000/book \
  -d "room_id=5" \
  -d "check_in=2026-05-01" \
  -d "check_out=2026-05-05" \
  -d "num_guests=2" \
  -d "special_requests=Late+check-in+around+10pm+please" \
  -d "csrf_token=YOUR_CSRF_TOKEN" \
  -b "PHPSESSID=your_session_id" \
  -L
```

**CSRF Required:** Yes
**Description:** Cancels a booking. Only bookings with status `pending` or `confirmed` can be cancelled. The booking must belong to the authenticated user unless the user is an admin (IDOR protection with admin override).

**URL Parameters:**

| Parameter | Type    | Required | Validation                                    |
|-----------|---------|----------|-----------------------------------------------|

**POST Fields:**

**Success Response:** 302 redirect to `/bookings` with flash message "Booking **{reference}** has been cancelled."

- Not logged in: 302 redirect to `/login?redirect=<current_uri>`.

| id        | integer | Yes      | Must be numeric (`is_numeric`). Must correspond to an existing, non-deleted user. Must not be the current admin's own ID. |

**POST Fields:**

| Field      | Type   | Required | Validation                                 |
|------------|--------|----------|--------------------------------------------|
| new_role   | string | Yes      | Must be one of: `admin`, `user`.           |
| csrf_token | string | Yes      | Must match session token.                  |

**Self-Protection:** If `id` equals the currently authenticated admin's user ID, the action is rejected with an error.

**Success Response:** 302 redirect to `/admin/users` with flash message "User role changed to {Role} successfully."

**Error Responses:**
- Not logged in: 302 redirect to `/login`.
- Not admin: 403 Forbidden.
- Invalid CSRF: 302 redirect to `/admin/users` with flash error "Invalid security token. Please try again."
- Invalid ID: 302 redirect to `/admin/users` with flash error "Invalid user ID."
- Self-role-change: 302 redirect to `/admin/users` with flash error "You cannot change your own role."
- Invalid role: 302 redirect to `/admin/users` with flash error "Invalid role value."
- User not found: 302 redirect to `/admin/users` with flash error "User not found."

**Example:**
```bash
curl -X POST http://localhost:8000/admin/users/role/7 \
  -d "new_role=admin" \
  -d "csrf_token=YOUR_CSRF_TOKEN" \
  -b "PHPSESSID=admin_session_id" \
  -L
```

## Appendix B: GST Pricing Rules

All prices are in Indian Rupees (INR).

| Room Base Price (per night) | GST Rate |
|-----------------------------|----------|
| >= 7,500                    | 18%      |
| < 7,500                     | 12%      |

**Calculation formula:**
```
nights      = check_out - check_in (in days)
base_total  = base_price_per_night * nights
tax_amount  = base_total * gst_rate
total_price = base_total + tax_amount
```

All amounts are rounded to 2 decimal places using PHP's `round()`.

## Appendix C: Security Notes

1. **CSRF Protection:** All POST endpoints validate a `csrf_token` field against `$_SESSION['csrf_token']` using `hash_equals()` (timing-safe comparison). This can be disabled in "vulnerable" mode for cybersecurity education.

2. **IDOR Prevention:** Booking views and cancellations verify that the booking belongs to the authenticated user. Admin role provides an override.

3. **Input Sanitization:** All user-provided text stored in the database is sanitized with `strip_tags()` to prevent stored XSS.

4. **Password Security (Secure Mode):** Passwords are hashed with bcrypt (cost factor 12). Sessions are regenerated on login. Accounts lock after 5 failed login attempts for 30 minutes.

5. **SQL Injection Prevention:** All database queries use parameterized prepared statements. Filter values are whitelist-validated before use.

6. **Email Enumeration Prevention:** The forgot-password endpoint always returns the same message regardless of whether the email exists.

7. **Admin Self-Protection:** Admins cannot deactivate their own account or change their own role.

8. **Password Column Exclusion:** The admin user listing explicitly excludes the `password` column from SELECT queries.
